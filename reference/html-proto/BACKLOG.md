# Backlog

This file is a parking lot for deferred work on the html-proto, not a session agenda. Items live here because they came up in past sessions, got considered, and were deferred for a reason — usually "later," "blocked on X," or "needs the user's call."

## How to use this file

- **Claude reads it for reference, not as a directive.** Don't open a session by attacking the backlog. Wait for the user to point at something. If the user hasn't mentioned the backlog, don't proactively start backlog work.
- **The user picks what to work on.** If you finish a task and have idle attention, surface 1-2 relevant backlog items as suggestions — don't just start the next one.
- **Update this file when items change state.** Adding a new deferred item, moving one to "Recently done," or marking one rejected — all part of normal work. Add the version or commit hash to "Recently done" entries so the history reads in order.
- **Prune "Recently done" periodically.** Once an item is a few versions old and visibly shipped, drop it. This isn't a changelog — git log is the changelog.
- **No version bump for editing this file.** It's not browser-served. Same rule as CLAUDE.md.

### Other

- **Collapse duplicated "duration" handling (dedupe, not a new framework)** — temporary effects are reverted by ~5 copy-pasted mechanisms in two shapes: the **EOT clock** (`tempPower`/`tempTou`, `eotGrants`, `typeGrants` with `eot:true`, `tempControlUntilEot` — all zeroed in the cleanup loop ~`engine.js:5733`) and the **leave-play listener** (`grantedBy` Map via `clearRestrictionsFromSource`, plus `typeGrants` with `eot:false`). The win is one shared EOT-revert helper + reusing the `grantedBy`-style leave-play listener, so the *next* temporary effect registers instead of adding a field + a cleanup site. **Do it when next working in combat/keyword-cleanup code anyway**, or when a card needs a *third* duration (e.g. "until your next turn") — that card is the moment it pays for itself. Deliberately do NOT build a general `until(<any event>)` registry: it'd serve zero current cards and is the "generic god-object event-matcher" the CLAUDE.md style notes warn against. No bug today; this is pure dedupe. (Considered + deferred this session; the longer spec was cut as premature.)
- **Make shared card-pick flows keyboard/accessibility friendly** — `renderCardPicker()` renders pickable card frames as clickable card elements, which is fine for mouse playtesting but weak for keyboard navigation, screen readers, and role-based browser tests. Low-risk first pass: wrap each pick in a real semantic control while preserving the current card visuals, add visible focus styling, and verify Enter/Space activation across boon, draft, post-draft, and in-game search pickers. Browser-only polish; no version bump needed for backlog-only tracking. Surfaced in the prototype UX/pass review.
- **SVG disc for {C}/{T}/{X}/numeric mana pips** — the 5 WUBRG pips are SVG (`assets/mana/{W,U,B,R,G}.svg`); the rest still use CSS letter-on-disc. Design an SVG disc treatment to match the WUBRG family (number content stays — by design).

These are normal browser/DOM edge cases and not really necessary to fix for the prototype; if debugging modal focus or fullscreen behavior gets annoying, change them to quiet `console.warn` breadcrumbs. (The 7 `try { render(); } catch (_) {}` repeats in `renderSettings` were retired in v1.0.182 — `render()` now deep-guards `!G || !G.you || !G.opp || !G.phase`.)

- **Decouple `SETTINGS_PANEL.show()` from `renderPanel()`** — `show()` currently always re-renders before opening the modal. Asymmetric coupling: callers can render-without-show but not show-without-render. Only one caller today (the gear button in `controller.js` init) and only one mode (open-fresh), so YAGNI. Note here so a future feature that needs to re-show without re-rendering (e.g. a keyboard shortcut while the modal is already open) has the API note already on file.
- **Reconsider `cardToViewModel`'s `opts` interface** — `opts.inHand` and `opts.overrideOracleText` are only set by `makeCardEl` (and `makeSyntheticCard` via passthrough). `openCardPopup` calls `cardToViewModel(card)` with no opts. The implicit contract works but isn't enforced. Possible shapes when revisiting: split into `cardToViewModel(card)` + `cardToViewModelInHand(card)`, or a `CardViewModel` class with `.forHand()` / `.forPopup()` methods. Not worth a refactor commit on its own; revisit when a third consumer appears.

- **Unify the two counter models (named-counter store vs +1/+1-as-stat)** — there are now two notions of "counter" reachable through the single `add_counter` effect, and they don't share storage. (1) **+1/+1 counters** aren't stored as counters at all: the v2.0 **D2** collapse folded them into permanent stat boosts (`pump duration:permanent` → `permPower`/`permTou`), and the `add_counter` +1/+1 branch now survives only for the runtime generators (`trigger-generator.js` `addCounterSelf` + Mercurial Adept's rolled "Standard-bearer" ability) — never card JSON. Despite zero static-card users the kind stayed fully wired — handler, AI valuation (scored like a permanent pump), a card-text case, and an empower-fields entry (`triggers.js`) — and `effect_migration_test` enforced that no card uses it (narrowed in v2.0.86 to forbid only the +1/+1 counter-less form). (2) **Named counters** (verse, v2.0.86) live in the new `card.counters:{}` map and never touch P/T. Latent wart: a +1/+1 counter can't be *removed* or *proliferated* as a counter, because it's indistinguishable from any other permanent stat boost — there's no discrete +1/+1 entry to act on. No bug today (no card needs discrete +1/+1). **Do it when a card first needs removable/proliferable +1/+1 counters** (proliferate, −1/−1, "remove a +1/+1 counter"): route +1/+1 through the same `counters` store (e.g. `counters['+1/+1']` summed into `getStats`) so every counter shares one model and `add_counter`/`remove_counters` work uniformly. Surfaced building Hymnwright's named-counter primitive (v2.0.86).

- **Unify the human cast and trigger target pick-loops (multi-slot selection)** — the two interactive multi-slot pickers are still parallel implementations: the **trigger** loop is `advanceTriggerTargetPrompt` (`engine.js`, slot-by-slot via `tsExcludePicked`), while the **cast** loop is `buildPendingActionWithTarget` / `pendingTarget.pickedSlots` in `controller.js` (dim-in-render + post-assembly `tsIsLegalSet`). PR #86 took the first step — the render cast-highlight now drops already-picked creatures through the shared `ENGINE.tsExcludePicked` (so the cross-slot *rule* lives once) — but the two pick *loops* remain separate, so future multi-slot UX (undo-pick, slot reorder, per-slot prompts on cast) has to be built in both or they drift. Do it when the next multi-slot UX feature lands. (Slice 4 of `docs/plans/plan-unified-target-selection.md`, deferred.)
- **`card-text.js` `_another` derivation is maintained separately from the engine's distinctness rule** — `describeEffectList` now decides "another target creature" from the `distinct_targets` flag + slot **target type** repetition (PR #86 fixed the old JSON.stringify-signature fragility), which matches the engine for every current card. But it's still a *parallel* derivation, not a read of the engine's actual per-pick legality. The two could diverge for an exotic future card (e.g. a distinct set spanning genuinely different object types that can still share one object, like `creature` + `permanent`). Revisit if/when such a card is added: drive the text from the same legality the engine enforces rather than a type-repetition heuristic. Surfaced in the PR #86 review.
